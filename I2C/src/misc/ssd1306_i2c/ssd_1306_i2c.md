### How it works

**I2C protocol for SSD1306** — every transfer has a 1-byte control prefix:

| Control byte | Meaning |
|---|---|
| `0x00` | The following bytes are **commands** |
| `0x40` | The following bytes are **pixel data** (GDDRAM) |

So `write(fd, buf, n)` where `buf = { 0x00, CMD_DISPLAY_ON }` is all it takes to send a command.

---

**Framebuffer layout** — the SSD1306 uses vertical bit packing:

```
page 0: rows 0–7   (bit 0 = row 0, bit 7 = row 7)
page 1: rows 8–15
...
page 7: rows 56–63
```

So `fb[page][col]` — `fb[y/8][x]`, with `bit = y%8`. That's why `fb_pixel()` does:
```c
fb[y / 8][x] |= (1 << (y % 8));
```

---

**Full flush in one write** — with horizontal addressing mode (`0x20, 0x00`) the display auto-increments column then page, so the entire 1024-byte framebuffer goes in a single `write()` call — well within the 8 KB kernel limit.

---

**Build & run:**
```bash
gcc -O2 -Wall -o ssd1306_i2c ssd1306_i2c.c

# default: /dev/i2c-1, address 0x3C
./ssd1306_i2c

# custom bus / alternate address
./ssd1306_i2c /dev/i2c-3 0x3D
```
